### Version 1.1.0

**Version:** 1.1.0 **Date:** 2026-05-26 **Environment:** Mainnet, Preprod, Preview

> **Required:** consumers must resolve `@midnight-ntwrk/ledger-v8` to `>=8.1.0`. The SDK's internal balancing flows
> depend on builder APIs (`Transaction.addIntent`, `Transaction.addZswapOffer`) that only exist on ledger 8.1.0 — older
> ledger versions will not work with this release.

---

### High-level summary

A maintenance release with two notable bug fixes in dust handling and a broader import surface on the barrel package.
The fee balancer no longer fails with `InsufficientFunds` on freshly-registered dust UTxOs (issue #383); Dust
registration and deregistration no longer race with concurrent transactions over the same Night UTxOs; the
`@midnight-ntwrk/wallet-sdk` barrel adds subpath exports for every remaining subpackage; the facade's
`wallet-sdk-abstractions` runtime dependency is correctly declared so Vite/esbuild stops failing on the missing
`TransactionHistoryStorage` export; and the SDK moves to ledger `8.1.0` — a required upgrade, with internal balancing
flows refactored onto the new builder API.

---

### Audience

This release is relevant for developers who:

- Build wallets for the Midnight network
- Integrate Midnight token transfers into their DApps
- Need to manage shielded (private) and unshielded token balances
- Implement atomic swaps between parties

### What changed (Summary of updates)

- **Required upgrade:** `@midnight-ntwrk/ledger-v8` `>=8.1.0`. Internal balancing in `dust-wallet`, `unshielded-wallet`,
  and `shielded-wallet` is refactored onto the new ledger builder API (`Transaction.addIntent`,
  `Transaction.addZswapOffer`); older ledger versions are incompatible.
- Fix `InsufficientFunds` from the dust fee balancer despite ample dust (#383); two compounding coin-selection defects
  resolved
- Fix race in Dust registration / deregistration that could double-use a Night UTxO already in flight in another
  transaction; chosen Night UTxOs are now booked (available → pending) at build time
- New methods to support split Dust-registration flows: `UnshieldedWallet.rotateUtxos`,
  `DustWallet.splitNightUtxosForDustRegistration`, `DustWallet.attachDustRegistration`
- Barrel `@midnight-ntwrk/wallet-sdk` adds subpath exports for `indexer-client`, `node-client`, `prover-client`,
  `runtime`, and `utilities`, plus every nested subpath those packages already expose, plus flat re-exports of
  `prover-client` and `utilities` and namespaced exports for `Capabilities`, `IndexerClient`, `NodeClient`, and
  `Runtime`
- Fix `@midnight-ntwrk/wallet-sdk-facade` declaring `wallet-sdk-abstractions` as a devDependency despite importing it at
  runtime — resolves Vite/esbuild dep-optimization failures
- Internal `@midnight-ntwrk/wallet-sdk-*` sibling dependencies widened from exact versions to caret ranges so consumers
  can dedupe to a single installed copy

### New features

**Additional subpath exports on `@midnight-ntwrk/wallet-sdk`**

**Description:** The barrel package now exposes `indexer-client`, `node-client`, `prover-client`, `runtime`, and
`utilities` as subpath entry points (each matching the package folder name), plus every nested subpath those packages
already expose (`/v1`, `/effect`, `/abstractions`, `/balancer`, `/pendingTransactions`, `/proving`, `/simulation`,
`/submission`, `/networking`, `/types`, `/testing`). The main entry point also re-exports `prover-client` and
`utilities` flat, and adds namespaced `Capabilities`, `IndexerClient`, `NodeClient`, and `Runtime` (namespaced to avoid
name collisions). Legacy `/proving` and `/testing` aliases remain for backwards compatibility.

---

**Split Dust-registration build helpers**

**Description:** To make Dust registration interact safely with other in-flight transactions, the build flow is split
into composable steps. New API:

- `UnshieldedWallet.rotateUtxos` — pre-spends and re-creates Night UTxOs so that registration has a known, isolated set
  of inputs to attach to.
- `DustWallet.splitNightUtxosForDustRegistration` — produces the dust-side intent fragments derived from a chosen set of
  Night UTxOs.
- `DustWallet.attachDustRegistration` — attaches the registration to the in-progress transaction.

These compose with the existing transaction-building APIs and are what makes the race fix possible; existing single-call
registration paths continue to work.

### Improvements

- Internal balancing flows in `dust-wallet`, `unshielded-wallet`, and `shielded-wallet` are refactored to use the ledger
  8.1.0 builder API (`Transaction.addIntent`, `Transaction.addZswapOffer`) instead of post-construction field mutation
  on `Transaction.fromParts(...)`. No public API changes.
- Internal `@midnight-ntwrk/wallet-sdk-*` sibling dependency ranges widened to caret. Consumers installing the SDK as
  multiple subpackages now resolve to a single deduplicated copy of each shared sibling rather than several pinned
  copies.
- Refreshed devDependencies (renovate-driven): `@effect/cluster`, `@graphql-codegen/*` to v6/v7, `tar-stream`; root
  resolutions for `protobufjs` v8, `immutable` v5, `undici` v8, `@substrate/connect` v2; CI `actions/setup-node` to
  v6.4.0. No published-package source changes.

### Breaking changes or required actions for developers

**Ledger `>=8.1.0` is required (all wallet packages)**

The SDK now depends on the ledger 8.1.0 builder API (`Transaction.addIntent`, `Transaction.addZswapOffer`) in its
internal balancing flows. Consumers **must** resolve `@midnight-ntwrk/ledger-v8` to `>=8.1.0`; pre-8.1.0 ledger versions
are incompatible with this release and the SDK will not function correctly against them. There are no public API changes
in the SDK itself from this bump.

---

**`CoinsAndBalances.CoinSelection` is no longer parametrized (`@midnight-ntwrk/wallet-sdk-dust-wallet`)**

As part of the dust fee-balancer fix, the local `CoinSelection` type and `chooseCoin` signature were realigned with the
capabilities `CoinRecipe` API. `CoinsAndBalances.CoinSelection` changed from `CoinSelection<TInput>` to a polymorphic
`CoinSelection` — type references must be updated. The previous wiring was non-functional (the variant's `coinSelection`
slot was never invoked), so any consumer who customized it had no observable behavior; only the type reference needs
updating.

### Fixed defect list

- Dust fee balancing failing with `InsufficientFunds` despite ample dust (issue #383). Two compounding coin-selection
  defects fixed: (1) the local `chooseCoin` now skips zero-value coins, so a freshly-registered dust UTxO with
  `generatedNow === 0` no longer wastes an iteration as the smallest candidate, and the variant now passes its
  configured coin selection through to `getBalanceRecipe` (the slot was previously never invoked); (2)
  `computeBalancingRecipe` identifies coins by `token.nonce` rather than by `value` when removing the just-picked coin
  from the working set — value-equality previously caused a single pick to drop the entire cohort of dust UTxOs sharing
  the same `generatedNow` (a routine outcome once their backing Night UTxOs reach `maxCap`), making most of the wallet's
  spendable dust invisible to the balancer.
- Race in Dust registration / deregistration where two in-flight transactions could pick the same Night UTxOs as inputs.
  The build flow now books the chosen Night UTxOs (available → pending) at build time, so a conflicting concurrent build
  fails immediately with `SpendUtxoError` rather than only at submission.
- `@midnight-ntwrk/wallet-sdk-facade` previously declared `@midnight-ntwrk/wallet-sdk-abstractions` as a devDependency
  despite importing it at runtime from `src/index.ts`. Consumers now correctly receive `wallet-sdk-abstractions` on
  install, resolving Vite/esbuild dep-optimization failures with
  `No matching export ... for import "TransactionHistoryStorage"`.
