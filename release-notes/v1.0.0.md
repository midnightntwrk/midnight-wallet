### Version 1.0.0

**Version:** 1.0.0 **Date:** 2026-04-23 **Environment:** Mainnet, Preprod, Preview

---

### High-level summary

First release under the new versioning scheme: release-note filenames now track
[`@midnight-ntwrk/wallet-sdk`](../packages/wallet-sdk) — the barrel package — rather than the facade. This release ships
that barrel at 1.0.0 as the recommended install path, and reshapes transaction-history across all three wallets onto a
single shared storage layer. The shielded and dust wallet's transaction history is **implemented** on the shared layer,
the unshielded wallet is **refactored** onto it. Dust coin getters now return richer `DustFullInfo` entries by default,
balancing-transaction segment placement becomes deterministic, the Simulator moves into `capabilities` with a proper
fee-payment and block-production model, and the SDK moves to ledger `8.0.3`.

---

### Audience

This release is relevant for developers who:

- Build wallets for the Midnight network
- Integrate Midnight token transfers into their DApps
- Need to manage shielded (private) and unshielded token balances
- Implement atomic swaps between parties

---

### What changed (Summary of updates)

- New `@midnight-ntwrk/wallet-sdk` barrel package — single install, one import surface, sub-path exports for each
  subpackage
- Shared transaction-history storage layer in `@midnight-ntwrk/wallet-sdk-abstractions`; shielded and wallet
  reimplemented on it, unshielded refactored onto it
- Dust coin getters (`availableCoins`, `pendingCoins`, `totalCoins`) return `DustFullInfo` with generation details
- `CustomDustWallet` factory and generic types exposed, matching the shielded/unshielded customisation pattern
- Deterministic balancing segment id (lowest unused fallible segment in `[1, 65535]`); `findAvailableSegmentId` helper
  added to dust and unshielded wallets
- Shielded balancing now supports multiple fallible segments
- `batchUpdates` option on `DefaultSyncConfiguration` for shielded and dust sync streams
- Simulator unified into `@midnight-ntwrk/wallet-sdk-capabilities` with proper fee-payment and block-production
  behaviour
- Ledger bumped to `8.0.3`
- `WalletFacade.prototype.signRecipe` no longer fails when a valid transaction has no intents

### New features

**Barrel package `@midnight-ntwrk/wallet-sdk`**

**Description:** New top-level package that re-exports every wallet SDK subpackage. Consumers install one package and
import either from the root (`@midnight-ntwrk/wallet-sdk`) or from the stable sub-paths (`/facade`, `/dust`,
`/shielded`, `/unshielded`, `/address-format`, `/hd`, `/capabilities`, `/proving`, `/testing`). The existing per-package
imports continue to work unchanged. See
[`packages/docs-snippets/src/snippets/initialization.ts`](../packages/docs-snippets/src/snippets/initialization.ts) for
the canonical usage.

---

**Dust `transactionHistory`**

**Description:** The dust wallet now records transaction history in the same shared storage layer used by the shielded
and unshielded wallets. The `WalletEntry` schema produced by the facade gains an optional `dust` section alongside the
existing `shielded` and `unshielded` sections.

---

**`CustomDustWallet` factory and generic types**

**Description:** `CustomDustWallet` factory, `CustomizedDustWallet` and `CustomizedDustWalletClass` generic types are
now exported from `@midnight-ntwrk/wallet-sdk-dust-wallet`. Variant builders can customise the dust wallet with the same
pattern already used for the shielded and unshielded wallets.

---

**Multi-segment shielded balancing**

**Description:** Shielded offers can now be balanced across multiple fallible segments within a single transaction,
lifting the previous one-segment constraint.

---

**`batchUpdates` sync option**

**Description:** `DefaultSyncConfiguration` accepts a new optional `batchUpdates` field controlling sync-stream batching
— size, timeout, and spacing between batches — for the shielded and dust sync streams.

---

**Deterministic balancing segment id**

**Description:** Balancing transactions are built with `Transaction.fromParts` + explicit
`intents.set(segmentId, intent)`, where `segmentId` is the lowest unused fallible segment in `[1, 65535]`. Replaces
`Transaction.fromPartsRandomized`, making segment placement reproducible. A `findAvailableSegmentId` helper is exported
from the dust wallet's `Transacting.ts` and exposed as a method on the unshielded wallet's `TransactionOps`.

### Breaking changes or required actions for developers

**Consolidated import surface via the barrel**

Existing per-package imports continue to work, but the recommended pattern is now a single import from the barrel. Call
sites that imported `InMemoryTransactionHistoryStorage` from `@midnight-ntwrk/wallet-sdk-unshielded-wallet` should
prefer the barrel or the shared `@midnight-ntwrk/wallet-sdk-abstractions` package; the symbol is the same.

```ts
// Before (Facade v3.0.0)
import { DustWallet } from '@midnight-ntwrk/wallet-sdk-dust-wallet';
import { type DefaultConfiguration, WalletFacade } from '@midnight-ntwrk/wallet-sdk-facade';
import { HDWallet, Roles } from '@midnight-ntwrk/wallet-sdk-hd';
import { ShieldedWallet } from '@midnight-ntwrk/wallet-sdk-shielded';
import {
  createKeystore,
  InMemoryTransactionHistoryStorage,
  PublicKey,
  UnshieldedWallet,
} from '@midnight-ntwrk/wallet-sdk-unshielded-wallet';

// After (v1.0.0 — one package)
import {
  type DefaultConfiguration,
  DustWallet,
  InMemoryTransactionHistoryStorage,
  WalletEntrySchema,
  WalletFacade,
  HDWallet,
  Roles,
  ShieldedWallet,
  createKeystore,
  PublicKey,
  UnshieldedWallet,
  mergeWalletEntries,
} from '@midnight-ntwrk/wallet-sdk';
```

---

**`InMemoryTransactionHistoryStorage` constructor signature changed (`@midnight-ntwrk/wallet-sdk-abstractions`,
`@midnight-ntwrk/wallet-sdk-facade`)**

The zero-argument constructor is removed. `InMemoryTransactionHistoryStorage` now requires a schema and a merge function
so that the storage layer can serialise, restore, and merge entries itself. Both arguments are exported from
`@midnight-ntwrk/wallet-sdk-facade` (and re-exported from the barrel).

```ts
// Before (v3.0.0)
const configuration: DefaultConfiguration = {
  // ...
  txHistoryStorage: new InMemoryTransactionHistoryStorage(),
};

// After (v1.0.0)
const configuration: DefaultConfiguration = {
  // ...
  txHistoryStorage: new InMemoryTransactionHistoryStorage(WalletEntrySchema, mergeWalletEntries),
};
```

The `WalletEntry` type is `{ shielded?, unshielded?, dust? }` plus the common transaction-history fields. The `dust`
section is new in this release. The schema composes `ShieldedSectionSchema`, `UnshieldedSectionSchema`, and
`DustSectionSchema` from the respective wallet packages.

---

**`WalletFacade.getAllFromTxHistory` returns a promise, not an async iterator (`@midnight-ntwrk/wallet-sdk-facade`)**

The method signature changed from `AsyncIterableIterator<WalletEntry>` to `Promise<WalletEntry[]>`.

```ts
// Before (v3.0.0)
for await (const entry of wallet.getAllFromTxHistory()) {
  handle(entry);
}

// After (v1.0.0)
const entries = await wallet.getAllFromTxHistory();
entries.forEach(handle);
```

---

**Dust coin getters return `DustFullInfo` (`@midnight-ntwrk/wallet-sdk-dust-wallet`)**

`availableCoins`, `pendingCoins`, and `totalCoins` on `DustWalletState` now all return `readonly DustFullInfo[]` (with
generation details) instead of raw `Dust`. The underlying `Dust` token is accessible via `.token` on each entry. The
corresponding capability methods `getAvailableCoins`, `getPendingCoins`, and `getTotalCoins` each accept an optional
`time` parameter that defaults to the wallet's `syncTime`. The previous `availableCoinsWithFullInfo(time)` method is
removed — it was the only "full-info" variant that actually existed, and its behaviour is now the default.

```ts
// Before (dust-wallet 3.x)
const coins: readonly Dust[] = dustWalletState.availableCoins;
const pending: readonly Dust[] = dustWalletState.pendingCoins;
const total: readonly Dust[] = dustWalletState.totalCoins;
const withInfo: readonly DustFullInfo[] = dustWalletState.availableCoinsWithFullInfo(new Date());

// After (dust-wallet 4.0.0)
const coins: readonly DustFullInfo[] = dustWalletState.availableCoins;
const pending: readonly DustFullInfo[] = dustWalletState.pendingCoins;
const total: readonly DustFullInfo[] = dustWalletState.totalCoins;

// Raw Dust tokens are available via `.token`:
const raw: readonly Dust[] = coins.map((c) => c.token);

// Optional `time` argument replaces the old `availableCoinsWithFullInfo(time)` call:
const atTime = dustWalletState.capabilities.coinsAndBalances.getAvailableCoins(dustWalletState.state, someDate);
```

---

**Shielded wallet `transactionHistory` implemented on shared storage (`@midnight-ntwrk/wallet-sdk-shielded`)**

The shielded-wallet transaction-history implementation is done with one that reads and writes through the new
`TransactionHistoryStorage` in `@midnight-ntwrk/wallet-sdk-abstractions`. The public-facing property on the wallet is
`transactionHistory`; the stored entry shape now follows `ShieldedSectionSchema` (exported from
`@midnight-ntwrk/wallet-sdk-shielded` and surfaced via `WalletEntrySchema.shielded`). Any custom serialisation or
restoration code that inspects entry shape must be re-verified.

---

**Unshielded wallet `transactionHistory` refactored onto shared storage
(`@midnight-ntwrk/wallet-sdk-unshielded-wallet`)**

The unshielded wallet's transaction-history is refactored onto the shared storage layer. Entry shape is now
`UnshieldedSectionSchema` (exported from `@midnight-ntwrk/wallet-sdk-unshielded-wallet` and surfaced via
`WalletEntrySchema.unshielded`). Re-verify any custom serialisation or restoration code that inspects entry shape.

---

**Ledger bumped to `8.0.3`**

Consumers must align `@midnight-ntwrk/ledger-v8` to `8.0.3`.
